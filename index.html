01

The governing rule

Downstream work begins only when the evidence needed to do it properly is present.

Do not analyse, test or code from assumptions when the product documentation, template or existing codebase can—and should—be supplied first.

Intake gate A

Requirement analysis

The Product Definition Agent first grounds the request in the actual product and process context.

  • Business requirement input
  • Product or process documentation
  • Related Jira stories or examples
  • Known scope, countries and products
Product documentation may be waived only with a documented Product Owner decision.
Intake gate B

Test-script creation

The Quality Agent must understand both expected behaviour and the format used by the delivery organisation.

  • Approved requirement and acceptance criteria
  • Relevant product documentation
  • Target test-script template
  • Regression examples or reference tests
A default internal template is allowed only after a documented Quality Lead waiver.
Intake gate C

Coding

The Development Agent works against the real repository, approved design and reproducible build environment.

  • Approved requirement
  • Approved solution design
  • Git repository, local path or code ZIP
  • Build and test commands
  • Coding standards and API documentation
Repository access, approved inputs, build command and test command cannot be waived.
02

Operating model

One orchestrator routes work through an intake gate to four stable specialist roles.

User · Jira · Product Owner
BTP@MA Orchestrator
Mandatory intake gate
1 Product Definition Agent Requirement logic, acceptance criteria, scope and traceability.
2 Solution Agent BTP architecture, integrations, APIs, data and security design.
3 Development Agent Implementation plan, isolated coding, tests and pull request.
4 Quality Agent Test design, independent review, regression and release evidence.
03

Delivery workflow

The process is designed to reduce interpretation gaps and prevent unverified completion claims.

Step 1Classify the requested workflow
Step 2Collect and validate required inputs
Step 3Route to the accountable specialist
Step 4Review and approve the output
Step 5Freshly verify before completion

What the framework enforces

  • Material already supplied is inspected before asking again.
  • Missing mandatory inputs block the workflow.
  • Deliverables carry requirement-to-test-to-code traceability.
  • Human approvals are explicit and policy-driven.
  • Coding uses an isolated branch or worktree where applicable.
  • An independent review precedes every completion claim.

What the framework prevents

  • Inventing product logic from a short user prompt.
  • Generating test scripts in an unknown template.
  • Writing code without the real codebase.
  • Embedding access tokens in prompts or repositories.
  • Calling a result “done” without fresh evidence.
  • Allowing agents to operate outside approved tool boundaries.
04

Inside the repository

The starter package separates agent behaviour, policy, executable controls and delivery templates.

skills/

Eight reusable `SKILL.md` workflows for intake, analysis, design, testing, implementation, review and verification.

agents/

Definitions for the orchestrator and the four fixed specialist agents.

config/

Intake rules, approval gates, agent registry and default-deny tool policy.

src/

Python reference implementation of the intake validator, router and connector interfaces.

schemas/

Machine-readable schemas for request manifests and end-to-end traceability.

templates/

Requirement, solution-design, implementation-plan, test-case and traceability templates.

requests/

Ready-to-run examples for blocked and ready workflow scenarios.

.github/

Continuous integration, pull-request controls and structured issue intake.

docs/

Architecture, workflow, installation, security and extension guidance.

05

Quick start

Install the CLI, validate the policies and run a blocked and ready intake example.

1. Install

python -m venv .venv # Windows PowerShell .venv\Scripts\Activate.ps1 # macOS / Linux # source .venv/bin/activate python -m pip install --upgrade pip pip install -e ".[dev]"

2. Validate the framework

btp-ma validate-policy pytest ruff check .

3. See a blocked requirement request

btp-ma check \ --workflow requirement-analysis \ --request requests/examples/requirement-analysis-missing-docs.yaml

4. Route a ready coding request

btp-ma check \ --workflow coding \ --request requests/examples/coding-ready.yaml

5. Generate a new intake manifest

btp-ma template \ --workflow test-script-creation \ --output request.yaml

07

Connection principle

Agents call controlled interfaces; secrets remain outside prompts, skills and committed files.

Implement real-system connectors behind an approved tool gateway. Resolve credentials at runtime through your enterprise secret or destination service.

Connect next

  • Jira project and issue-type mappings
  • Product-documentation retrieval
  • GitHub Enterprise repository operations
  • SAP BTP deployment and log retrieval
  • Model routing, cost and data policies

Never commit

  • Personal API tokens
  • Passwords or client secrets
  • Production credentials
  • Private keys
  • Confidential document content in skill files
